## 四、C-3-ext freeze diff 增量模式

### 4.1 6 项核心能力

| § | 核心函数 | 能力 |
|---|---------|------|
| 1 | `_load_freeze_doc(version)` | 读 freeze 文档(v1.5.X.{md,json}),缺文件抛 `FileNotFoundError` |
| 2 | `_endpoint_key(ep)` | `(path, method)` 元组,method 自动大写 |
| 3 | `_extract_endpoints(doc)` | 兼容 2 种 doc 结构(ep list + OpenAPI paths dict) |
| 4 | `diff_freezes(prev, curr)` | 3 类变更检测(added / removed / changed) |
| 5 | `write_diff_reports(diff)` | JSON + MD 双格式报告落地 |
| 6 | `run_diff_mode(prev, curr)` | 主入口,返 0(成功)/ 2(文件缺失) |

### 4.2 工具调用

```bash
# 默认全量校验(m8t1 子进程)
py -m scripts.freeze_check

# 增量 diff 模式
py -m scripts.freeze_check --diff --prev v1.5.4 --curr v1.5.5

# JSON 报告
py -m scripts.freeze_check --diff --prev v1.5.4 --curr v1.5.5 \
    --report-json docs/freeze-diff-report-v1.5.4-to-v1.5.5.json

# MD 报告
py -m scripts.freeze_check --diff --prev v1.5.4 --curr v1.5.5 \
    --report-md docs/freeze-diff-report-v1.5.4-to-v1.5.5.md
```

### 4.3 退出码语义

- `0` — diff 全部成功
- `2` — freeze 文档不存在(prev 或 curr)
- `1` — 校验失败(保留原 freeze_check 模式语义)

### 4.4 3 类变更检测

| 类型 | 含义 | 检测方式 |
|------|------|---------|
| added | curr 有但 prev 没有的端点 | `(path, method)` 不在 prev 端点集合中 |
| removed | prev 有但 curr 没有的端点 | `(path, method)` 不在 curr 端点集合中 |
| changed | (path, method) 都存在但 summary 字段不一致 | `prev_summary != curr_summary` |

### 4.5 双 doc 结构兼容

| 结构 | 来源 | 提取逻辑 |
|------|------|---------|
| endpoints list | freeze_check 自身生成的 freeze doc | 直接遍历 list,`{path, method, summary}` |
| OpenAPI paths dict | 外部 OpenAPI 规范 | 遍历 `paths`,每个 path 下提取 method + summary |

## 五、C-6-ext self_test_harness 增强

### 5.1 3 项新核心能力

| 能力 | 实现 | 用途 |
|------|------|------|
| ThreadPoolExecutor 并行 | `_run_harness_parallel(scripts, timeout, workers)` | workers=1 串行(向后兼容),>1 启用并行,`as_completed` 按输入顺序取结果 |
| 多格式报告 | `format_html_report` + `format_csv_report` + 原 `format_json_report` | `--output-format {json,html,csv}` + `--report-html` / `--report-csv` |
| fail-fast 模式 | `run_harness(..., fail_fast=False)` | 第一个失败立即停(串行路径抛异常 + 并行路径直接 cancel 未启动 future) |

### 5.2 工具调用

```bash
# 默认全量(串行 + JSON 报告)
py -m scripts.self_test_harness

# 并行跑(4 worker)
py -m scripts.self_test_harness --workers 4

# 多格式报告(JSON + HTML + CSV)
py -m scripts.self_test_harness --report-json docs/harness-report.json
py -m scripts.self_test_harness --output-format html --report-html docs/harness-report.html
py -m scripts.self_test_harness --output-format csv --report-csv docs/harness-report.csv

# fail-fast 模式(第一个失败立即停)
py -m scripts.self_test_harness --fail-fast

# 超时控制(单脚本 timeout)
py -m scripts.self_test_harness --timeout 300
```

### 5.3 嵌套死锁防御(沿用 V1.5.7 三重)

1. **聚合型自动跳过**(`AGGREGATOR_PATTERNS`):避免 harness 调 m8t1 → m8t1 调 m9t1 → m9t1 调 harness 死循环
2. **`--skip-self` 选项**:排除 m16t3 自身测试,避免 m8t1 调 m16t3 → m16t3 调 m8t1 嵌套
3. **freeze_check §9 调 m8t1**:在 m15t1 / m16t4 测试中只检查函数签名不真跑,避免嵌套

### 5.4 XSS 防护

`format_html_report` 用 `html.escape()` 转义所有用户输入(脚本名 / 错误信息 / 测点详情),防止 XSS 注入。
`format_csv_report` 用 `csv.writer` + `io.StringIO` 正确处理包含逗号 / 引号 / 换行的字段。

### 5.5 顺序保持策略

并行路径用 `concurrent.futures.as_completed()` 收集 future,然后按输入 `scripts` 列表的索引顺序取结果,保证:
- 报告输出顺序与 CLI 输入顺序一致
- 测点编号稳定,便于 CI 日志对比
- fail-fast 时立即停,不依赖执行顺序
