## 二、P0 必填(7 项)详细说明

### 2.1 `DATABASE_URL`(PostgreSQL DSN,asyncpg)

**格式**:
```
postgresql+asyncpg://USER:PASSWORD@HOST:5432/DBNAME
```

**获取方式**:
- Supabase:Settings → Database → Connection string → URI(选 asyncpg 模式)
- Render Postgres:Internal Database URL
- 自托管:`postgresql+asyncpg://hunter:STRONG_PASSWORD@db.internal:5432/hunter_radar`

**沙箱 fallback**:`postgresql+asyncpg://hunter:hunter@localhost:5432/hunter_radar`(本机 docker postgres)

**生产必填性**:✅ 必填,无 fallback。生产 PG 16 需 `CREATE EXTENSION IF NOT EXISTS "uuid-ossp"`。

**校验**:`psql $DATABASE_URL -c '\dx'` 应含 uuid-ossp / pgcrypto / pg_trgm 三个扩展。

### 2.2 `DATABASE_URL_SYNC`(同步 DSN,Alembic + ETL)

**格式**:
```
postgresql+psycopg2://USER:PASSWORD@HOST:5432/DBNAME
```

**获取方式**:同 2.1,driver 换 `psycopg2`。

**用途**:
- Alembic 迁移
- BD-085 ETL 同步 batch 写入
- pytest fixture 初始化 schema

**生产必填性**:✅ 必填。

### 2.3 `SECRET_KEY`(JWT 签名密钥)

**格式**:32+ bytes 随机字符串

**生成方式**:
```bash
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

**沙箱 fallback**:`dev-only-change-me-in-prod-32-bytes-min`(M5 锁定,严禁生产使用)

**生产必填性**:✅ 必填,需 ≥32 bytes 且与历史值不同(避免旧 token 可伪造)

**校验**:长度 ≥32,不与 `dev-only-` 前缀。

### 2.4 `STRIPE_SECRET_KEY`(Stripe API 密钥)

**格式**:`sk_live_...`(生产) / `sk_test_...`(测试)

**获取方式**:Stripe Dashboard → Developers → API keys → Secret key

**生产必填性**:✅ 必填,生产必须用 `sk_live_` 前缀。

**校验**:前缀 `sk_live_` 或 `sk_test_`(后者仅 staging)。

### 2.5 `STRIPE_WEBHOOK_SECRET`(Webhook 签名密钥)

**格式**:`whsec_...`

**获取方式**:
1. Stripe Dashboard → Developers → Webhooks → Add endpoint
2. Endpoint URL:`https://api.hunter-radar.com/api/v1/subscriptions/webhook`
3. 监听事件:`customer.subscription.created/updated/deleted`、`invoice.payment_succeeded/failed`、`checkout.session.completed`
4. 创建后 Reveal signing secret → 复制 `whsec_...`

**沙箱 fallback**:未设时返 `signature_mode=sandbox_skip` + 200(显式标注,绝不 mock 200 伪装)

**生产必填性**:✅ 必填,缺则 webhook 永远 sandbox_skip,等同于关闭签名校验。

**校验**:前缀 `whsec_`,长度 ≥32。

### 2.6 `STRIPE_PRICE_PRO_MONTHLY`(Pro 月付 Price ID)

**格式**:`price_...`

**获取方式**:Stripe Dashboard → Products → Hunter Radar Pro → Pricing → Monthly → 复制 Price ID

**生产必填性**:✅ 必填,缺则 `POST /subscriptions/checkout` 报 500。

### 2.7 `STRIPE_PRICE_PRO_YEARLY`(Pro 年付 Price ID)

同 2.6,选 Yearly。

## 三、P1 必填(8 项)详细说明

### 3.1 `VAPID_PRIVATE_KEY` / `VAPID_PUBLIC_KEY`(WebPush 密钥对)

**格式**:
- `VAPID_PRIVATE_KEY`:base64url 编码的 EC 私钥(~44 字符)
- `VAPID_PUBLIC_KEY`:base64url 编码的 EC 公钥(~88 字符)

**生成方式**:
```bash
# 一次性生成,生产密钥必须不同于沙箱
python -c "
from py_vapid import Vapid
v = Vapid()
v.generate_keys()
print('VAPID_PRIVATE_KEY=' + v.private_pem())
print('VAPID_PUBLIC_KEY=' + v.public_key_urlsafe_base64())
"
```

**沙箱 fallback**:未设时 `push.py` 返 `push_skipped=true`(不 mock 200 伪装)

**生产必填性**:✅ 必填,缺则 PWA 用户永远收不到推送。

**校验**:
- 私钥长度 ≥80(包含 PEM 头尾)
- 公钥能 base64url decode 为 65 bytes EC 密钥
- 公私钥配对(`vapid.curve == cryptography.hazmat.primitives.asymmetric.ec.SECP256R1`)

### 3.2 `VAPID_CLAIMS_EMAIL`(WebPush claims 邮箱)

**格式**:RFC 5322 邮箱

**生产推荐**:`mailto:push@hunter-radar.com` 或 `mailto:ops@hunter-radar.com`

**沙箱 fallback**:`admin@hunter-radar.example`

**生产必填性**:✅ 必填,虽 sandbox 有 default,但生产应使用真实邮箱以通过 Push Service 验证。

### 3.3 `SENTRY_DSN`(前端 Sentry DSN)

**格式**:`https://PUBLIC_KEY@oXXXXXX.ingest.sentry.io/PROJECT_ID`

**获取方式**:Sentry.io → Project → Settings → Client Keys (DSN) → 复制

**沙箱 fallback**:未设时前端不初始化 Sentry(不 mock)

**生产必填性**:✅ 必填,缺则前端错误完全不可见。

**校验**:URL 包含 `sentry.io` + `/ingest/` 路径。

### 3.4 `ENV`(环境标识)

**取值**:`production` / `staging` / `development`

**生产必填性**:✅ 必填,且**必须等于** `production`(M5 锁定,`is_production` property 强依赖此值)

**校验**:值 == `production`(大小写不敏感)。

### 3.5 `LOG_LEVEL`

**取值**:`DEBUG` / `INFO` / `WARNING` / `ERROR`

**生产推荐**:`INFO`(避免 DEBUG 泄露敏感数据;ERROR 太少,WARNING 太多)

**沙箱 fallback**:`INFO`

### 3.6 `DEBUG`

**生产必填性**:❌ 必须显式设为 `false`(防止堆栈泄露)

**校验**:值 == `false`。

### 3.7 `CORS_ORIGINS`

**格式**:JSON list 字符串,例如 `["https://hunter-radar.com","https://www.hunter-radar.com"]`

**生产必填性**:✅ 必填,需包含生产域名;严禁 `*`(CSRF 风险)

**校验**:JSON 解析成功 + 含 `https://` + 不含 `*` + 不含 `localhost`(生产)。

## 四、P2 建议(7 项)

### 4.1 `REDIS_URL`

**格式**:`redis://USER:PASS@HOST:6379/0` 或 `rediss://...(TLS)`

**沙箱 fallback**:`redis://localhost:6379/0`

**生产建议**:Render Redis / Upstash / ElastiCache。

**降级**:连接失败时降级到 no_cache(R-09 已记录)。

### 4.2 `GOOGLE_OAUTH_CLIENT_ID` / `GOOGLE_OAUTH_CLIENT_SECRET`

**获取方式**:Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client IDs

**沙箱 fallback**:未设时跳过 Google OAuth,只支持 magic link。

**生产建议**:建议设,降低 magic link 邮件抵达率依赖。

### 4.3 `FINRA_SHORT_URL` / `SEC_EDGAR_BASE` / `SEC_USER_AGENT`

**用途**:m5t1 / m5t8 数据源配置

**生产必填性**:❌ 沿用默认即可,除非 FINRA / SEC 改 API。

**注意**:`SEC_USER_AGENT` 必须含邮箱(SEC 合规要求),默认 `HunterRadar/1.4 (ops@hunter-radar.example)`。

## 六、生产环境配置顺序(7 步)

### Step 1:Secrets 注入(Stripe / VAPID / Sentry)

按平台不同:

**Vercel**:
```bash
vercel env add STRIPE_SECRET_KEY production
vercel env add STRIPE_WEBHOOK_SECRET production
vercel env add STRIPE_PRICE_PRO_MONTHLY production
vercel env add STRIPE_PRICE_PRO_YEARLY production
vercel env add VAPID_PRIVATE_KEY production
vercel env add VAPID_PUBLIC_KEY production
vercel env add VAPID_CLAIMS_EMAIL production
vercel env add SENTRY_DSN production
vercel env add SECRET_KEY production
```

**Render / Railway**:
- Dashboard → Environment → Add Environment Variable
- 逐项添加(8 个 P0/P1 secrets)

**自托管**:
- `/etc/hunter-radar/.env` 权限 `chmod 600`
- 用 `envdir` / `direnv` 注入

### Step 2:Database 切换

- 创建生产 PG 16 实例(Supabase / Render / RDS)
- 跑 Alembic:`alembic upgrade head`(m5t9 已写 migration)
- 验证扩展:`psql $DATABASE_URL -c "SELECT extname FROM pg_extension;"` 应含 uuid-ossp / pgcrypto / pg_trgm

### Step 3:Redis 切换

- 创建生产 Redis 实例
- 验证 ping:`redis-cli -u $REDIS_URL ping` → PONG
- 沙箱降级:R-09(连接失败自动 no_cache,需 ops 监控)

### Step 4:VAPID 密钥生成

- 沙箱 VAPID 密钥对 **严禁** 用于生产(M3 落地的 `keys/vapid_private.pem` 是 dev-only)
- 生产必须用新生成的密钥对
- 私钥备份到 1Password / Bitwarden,公钥发布到前端 `frontend/src/lib/push.ts`

### Step 5:BD-086 双签替换

- 31 事件 `review_mode: sandbox_stub` → `prod_signoff_v1`
- 走 CR + 产品评审流程(BD-086 audit log)
- 替换前确保双签 reviewer 已注册

### Step 6:BD-085 ETL 切换

- `backtest_dataset_real.py`(沙箱) → `backtest_dataset_pg.py`(生产,asyncpg + PG)
- 跑全量回填:`python -m backend.scripts.backfill_pg --start 2024-01-01`
- 验证数据完整性:行数 = 4220(沙箱基线)

### Step 7:BD-087 v3.0-final 真实环境重测

- 切真实 PG 后重跑 `backtest_v30_final.py`
- scipy.stats.mannwhitneyu 替换 `_mann_whitney_u` 简化版
- 验证 Mann-Whitney U p ≥ 0.05 不显著 → 保持 v1.0 默认权重

## 八、回滚方案(env 缺失时降级)

| 缺失变量 | 系统行为 | 严重性 |
|---|---|---|
| `DATABASE_URL` | 启动失败,500 | P0 critical |
| `SECRET_KEY` | 启动失败,500 | P0 critical |
| `STRIPE_SECRET_KEY` | checkout 报 500 | P0 critical |
| `STRIPE_WEBHOOK_SECRET` | webhook 返 sandbox_skip + 200(显式标注) | P0 warn(等同关闭签名) |
| `STRIPE_PRICE_PRO_*` | checkout 报 500 | P0 critical |
| `VAPID_PRIVATE_KEY` | push 返 push_skipped=true | P1 warn(PWA 收不到推送) |
| `SENTRY_DSN` | 前端不初始化 Sentry | P1 warn(前端错误不可见) |
| `REDIS_URL` | 缓存降级 no_cache | P2 性能下降 |
| `GOOGLE_OAUTH_*` | 跳过 Google OAuth,magic link 仍可用 | P2 用户体验下降 |

**回滚 SOP**:
1. 健康检查 `GET /api/v1/health` 返 200 + `database: ok` + `redis: ok` + `stripe: ok`
2. webhook 签名模式检查:`POST /api/v1/subscriptions/webhook` 返 `signature_mode: prod_verified`
3. push 订阅检查:`POST /api/v1/push/subscribe` 返 `subscribed: true`(无 push_skipped)
4. 前端 Sentry:`window.Sentry` 已初始化 + DSN 匹配生产

## 十二、本文记忆(15 条)

1. P0 7 项缺失将直接 500 启动失败,P1 8 项缺失将 silent degradation
2. `STRIPE_WEBHOOK_SECRET` 未设时签名永远 sandbox_skip,等同关闭签名校验
3. `VAPID_PRIVATE_KEY` 沙箱密钥(`keys/vapid_private.pem`)严禁生产使用
4. `SENTRY_DSN` 缺失时前端 Sentry 静默失败,需 ops 监控
5. `ENV=production` 强依赖,`is_production` property 决定 webhook 签名模式
6. `CORS_ORIGINS` 严禁 `*` 或 `localhost`(生产),仅 https://
7. P2 7 项降级不致命,P3 9 项沿用默认
8. 配置顺序:secrets → database → redis → vapid → 双签 → ETL → 重测
9. 沙箱 fallback 显式标注(signature_mode / push_skipped),绝不 mock 200 伪装
10. VAPID 密钥对必须生产单独生成,与沙箱不同
11. SEC_USER_AGENT 必须含邮箱(SEC 合规要求)
12. Stripe webhook endpoint URL 需在 Stripe Dashboard 配置:`/api/v1/subscriptions/webhook`
13. 沙箱回滚 SOP:health → webhook signature_mode → push subscribed → Sentry DSN
14. 落地检查 12 项中 CI 自动 10 项 + 人工 2 项
15. V1.5 freeze 后由 `env_check.py` 接管 CI 校验
